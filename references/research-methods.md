# 素材补全策略

## 核心原则

Brief 不堆信息，只挑"写这篇需要的"。优先用用户给的东西，联网搜索是增强不是必须。

## 3 种来源（按优先级）

### 来源 1：用户直接提供（最优先）

用户可能提供：
- 一段素材链接
- 一份 topic-pool.md 中的选题条目
- 个人经历或观点

直接从中提炼：观点、痛点、案例、数据。

### 来源 2：联网搜索（可选增强，不是硬依赖）

当用户提供的信息不足以支撑 3 条素材时，可以联网补充。

搜索优先级：
1. **观点补充**：搜其他人的看法，找争议点
2. **案例补充**：搜具体案例、实践经历
3. **数据补充**：搜支撑论点的数据
4. **反方观点**：搜反面论据，增加冲突感

可用工具（按可用性选择，全部失败也不阻塞）：
- WebSearch
- `opencli twitter search --query "<关键词>" --limit 10`
- `opencli xiaohongshu search --keyword "<关键词>" --limit 10`
- `opencli zhihu hot --limit 20`

### 来源 3：知识推断（兜底）

联网也不够或不可用时：
- 从读者画像推断常见痛点
- 使用行业通用框架（SCQA、五力模型等）
- 从话题本身推断争议性

## 降级路径

| 场景 | 策略 |
|------|------|
| 用户给了充足素材 | 不联网，直接整理 |
| 素材不够但能联网 | 联网补齐到 3 条 |
| 素材不够且无法联网 | 知识推断补齐，在 brief 中标注"待验证" |
| 用户明确说"不要联网" | 只用用户提供的信息 + 知识推断 |

## 竞品分析（搜索时顺便做）

在搜索过程中，关注同类文章：

| 竞品特征 | 我们的机会 |
|---------|-----------|
| 全是清单体 | 写深度指南 |
| 内容过时 | 写最新版本 |
| 泛泛而谈 | 写实操+案例 |
| 角度单一 | 写反方或补充角度 |

## 素材质量门槛

每条素材必须通过：

| 检查项 | 通过标准 |
|--------|---------|
| 相关性 | 和选题直接相关 |
| 具体性 | 有具体细节，不是"很多人都在用" |
| 可验证 | 数据有来源，案例有时间/人物 |
| 冲突感 | 能引发讨论或不同意见 |
